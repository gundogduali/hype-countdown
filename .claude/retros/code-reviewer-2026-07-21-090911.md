## Real bug found and fixed
The new "DP-4: a timer emoji known to work renders true color pixels" test's `regionHasColor`
helper checked "does any pixel in a fixed region have R/G/B channels differing by more than 20"
as a proxy for "the emoji rendered in real color." This is unsound: the card's background is a
purple radial glow blended over near-black, and that glow *alone*, with **zero glyph drawn at
all**, already pushes every single pixel (28000/28000, verified with a standalone probe script)
in the exact sampled region past that same threshold. The test could never fail — it would pass
identically whether the emoji rendered in true color, as a flat monochrome glyph, or wasn't drawn
at all. This directly matches the review brief's own warning ("could false-pass on a non-color
glyph... due to anti-aliasing") except the actual mechanism was worse: it wasn't a narrow
anti-aliasing edge case, the *entire* region was already over-threshold from the background alone.

Fix: replaced the absolute-threshold check with a **differential** one. The test now renders the
same timer twice — once with the real emoji, once with the emoji swapped for a plain ASCII `.`
(same background/layout/watermark, no color glyph) — and diffs the two PNGs in the sampled
region. Subtracting cancels out the shared background exactly, so only the glyph's own
contribution remains; empirically verified this correctly reports ~198 max channel-spread for a
real color emoji glyph and exactly 0 for two identical renders (added as its own regression/sanity
test so this exact false-positive can't silently return). Files: `backend/test/og.test.js`.

## Lessons learned (added to my own agent file's Lessons Learned section)
- When reviewing a "pixel samples confirm behavior X" test, don't just read the threshold logic —
  actually render the *negative* case (the feature deliberately disabled, or literally nothing
  drawn) through the same helper and confirm the assertion can fail. I found this bug precisely
  because I did that (rendered background-only, got `hasColor: true`, with 100% of sampled pixels
  over threshold) instead of trusting that a "read the code, it looks like a real check" pass was
  enough.
