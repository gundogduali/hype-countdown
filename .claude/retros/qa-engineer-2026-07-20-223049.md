# QA self-correction retro — 2026-07-20 22:30

## What I reviewed
Re-examined my own SC-6/HM-6 work (report already delivered to the orchestrator) against
its own acceptance criteria and my own "tests must be runnable via `npm test`, twice in a
row" mandate, by actually re-running the real `npm test` chain end to end from a fresh
DB rather than trusting my earlier individual test-file runs.

## What I found (a real bug I had shipped)
My two additions to `frontend/e2e/messages.mjs` (newest-first-ordering check,
rate-limited-UI check) each called `createTimer()` — i.e. a real `POST /api/timers`.
`frontend/package.json`'s `"test"` script chains `smoke.mjs && reactions.mjs &&
messages.mjs` against **one** backend/DB, and `POST /api/timers` shares a single
20/hour-per-IP quota across all three scripts. I had only verified my additions against
`messages.mjs` run **in isolation** (with its own fresh DB) — never the actual chained
`npm test` invocation from a single fresh DB, which is what "runnable via `npm test`"
actually means.

Running the real chain from a fresh DB reproduced an **uncaught crash**
(`TypeError: Cannot read properties of undefined (reading 'slug')` inside `createTimer`,
because the 21st-and-later `POST /api/timers` calls return a `429` body, not
`{timer:{slug}}`), partway through `messages.mjs`. Worse: precise empirical measurement
(`smoke.mjs` + `reactions.mjs` alone consumed 10 of the 20 slots) showed this was
**already a pre-existing, latent bug** in the suite before I touched it — the original
(unmodified) `messages.mjs` needed 11 `createTimer()` calls, 1 more than the 10 left —
my 2 additions just made an already-broken chain fail earlier/harder.

## What I fixed now
Reduced `messages.mjs`'s total `createTimer()` calls from 11 down to 6, by reusing
already-created, state-compatible timers instead of creating a dedicated one per check:
- Newest-first-ordering check: folded into the existing "post + persist" block's timer
  (posts a 2nd message to the same slug) instead of a separate `createTimer()` call.
- Rate-limited-UI check: switched to the always-present curated `gta-6` slug — the
  message rate limit is keyed per-IP only (not per-slug), so which timer is used doesn't
  matter, and this check already required a fresh backend for re-runs regardless.
- The 5-case moderation-error-code loop (API level): switched from one fresh timer per
  case to a single shared timer for all 5 — every one of those submissions is rejected
  and never stored, so there's no cross-case state to corrupt.
- Mobile-width rendering check: reuses the desktop empty-state check's already-created,
  still-message-free slug instead of its own `createTimer()` call.

Verified the fix by running the **actual chained sequence** (`smoke.mjs` then
`reactions.mjs` then `messages.mjs`, in that order, against one fresh `DB_PATH`) twice in
a row from scratch: both times, all three scripts exited 0 with `RESULT: ALL CHECKS
PASSED` and zero `FAIL` lines. Also re-ran the backend `npm test` suite (141/141, twice)
and `node --check` on the edited file to confirm no syntax errors.

## What the orchestrator needs to know
- The fix is scoped entirely to `frontend/e2e/messages.mjs` (test-file changes only, no
  product code touched).
- The underlying constraint (all `POST /api/timers`-creating e2e scripts share one
  20/hour-per-IP quota when chained via `npm test`) is still fragile in principle — it
  happened to have exactly enough headroom in my fix, but any future e2e script addition
  that creates several more custom timers could tip it over again. Worth a note for
  whichever agent next touches these e2e scripts: prefer reusing an existing/shared
  timer over `createTimer()` where the test doesn't specifically need an isolated one.
- My final report to the orchestrator (already delivered) is otherwise unchanged — all
  SC-6/HM-6 acceptance criteria results, findings, and file paths in that report still
  stand; this correction only touched the persistent-test implementation, not the
  verdicts.
