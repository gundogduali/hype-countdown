# Retro — SC-5 (server-side social share meta tags on `/t/:slug`)

## What I did
- Added `backend/src/services/htmlTemplate.js`: `escapeHtml()` + `renderTimerHtml(indexHtml, timer, { pageUrl, imageUrl })`, which injects/replaces `<title>`, `<meta name="description">`, `og:title`/`og:description`/`og:image`/`og:url`/`og:type`, and `twitter:card=summary_large_image` into an in-memory HTML string.
- Modified the `if (staticDir)` SPA-fallback block in `backend/src/app.js`: reads `index.html` into memory once at startup (`indexHtmlTemplate`), and the `app.get('*', ...)` handler now special-cases paths matching `/^\/t\/([^/]+)\/?$/` (mirrors the frontend router's `/t/:slug` route in `frontend/src/App.jsx`). A known slug (via the existing `TimerService.getBySlug` — no duplicated lookup logic) gets the templated variant with an absolute `og:image` URL pointing at the already-shipped `GET /api/timers/:slug/og-image.png` (SC-3/SC-4); an unknown slug or any other path falls through unchanged to the original `res.sendFile(indexHtml, ...)`.
- Updated `docs/api.md` (v2.4 → v2.5): documented `GET /t/:slug` as a non-JSON, production-only contract addition, describing exactly which tags are injected and the escaping guarantee.
- Added a new `describe` block to `backend/test/static.test.js` (SC-5): known-slug meta tags + absolute-URL assertions, HTML-injection-escaping with a crafted `<script>`/`&`/`"` title, unknown-slug fallback, and a byte-identical-response check for `/`, `/create`, `/no/such/page`.
- Fixed the existing `static.test.js` "SPA fallback: unknown non-/api GET paths" test, which used `/t/gta-6` as a stand-in for "any unmatched SPA path" — `gta-6` is a real seeded curated slug, so this feature necessarily changes its response. Swapped that specific case for `/t/no-such-timer-slug` so the test still asserts what it originally intended (an unmatched path gets the plain static file) without asserting something now false.
- Verified live: built the frontend (`vite build`), served it with `STATIC_DIR` + a temp `DB_PATH`, and curled `/t/gta-6` (correct og/twitter tags, absolute `og:image` resolving to a real `200 image/png`), a POST-created custom timer with a `<script>`/`&`/`"quotes"`/`<b>` title (escaped correctly, no raw markup in the response), an unknown slug (unchanged plain HTML), `/` and `/create` (byte-identical via `diff` against the built `dist/index.html`), a non-GET to `/t/gta-6` (still JSON `404 not_found`, unaffected), and local dev with `STATIC_DIR` unset (existing "static serving disabled" test, unaffected).

## What I found
- The task brief's premise that `index.html` was "already loaded once at startup" was actually wrong for this codebase: the existing `indexHtml` variable in `app.js` was just a *path* string used with `res.sendFile` (which streams from disk on every request), not cached file content. I added the actual `readFileSync` cache myself rather than assuming it existed — worth flagging since blindly trusting the brief's description here would have led me to skip caching entirely and re-read the file on every `/t/:slug` request.
- `gta-6` (and every other curated seed slug) is a real timer in the default-seeded DB used by the existing static-serving tests, so this feature is a behavior change for that specific existing test case, not just new coverage. Had to actually re-read and fix the pre-existing test rather than only append new tests, per the acceptance criterion "full test suite passes."
- A `**/*` style block-comment (`og:*/twitter:*`) in a new JSDoc comment in the test file closed the comment early at the embedded `*/` and broke the whole file's syntax — caught immediately by running the suite, fixed by rewording (not repeating this specific literal-`*/` mistake going forward).

## What I fixed
- Added the missing `readFileSync` cache for `index.html` (`indexHtmlTemplate`) so the templating path never re-reads from disk per request.
- Corrected the existing static-serving test's `/t/gta-6` case to a non-existent slug so it still tests what it always intended.
- Reworded the JSDoc comment that contained a literal `*/` inside `og:*/twitter:*`.
- 138/138 tests pass (`node --test "test/*.test.js"`); live curl verification against a real built frontend covered every acceptance-criterion scenario (known slug, escaped injection attempt, unknown slug, byte-identical unrelated paths, non-GET, absolute og:image URL that actually resolves, local dev unaffected).

## Lesson
No new repeatable-mistake pattern beyond ones already recorded (orphan-process check was performed: my own temp verification server, started with a temp `DB_PATH`, was confirmed killed via `ps`; a separate, pre-existing `node --watch src/server.js` on port 3001 was left untouched since I never started it and it isn't safe to assume ownership of another concurrent agent's process in this multi-agent workspace).
