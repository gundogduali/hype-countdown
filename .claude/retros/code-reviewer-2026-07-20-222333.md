## Findings reported (not fixed — architectural/ambiguous per Rule 2)
- One **probable, low-severity** finding: `message_too_long`'s ratified copy ("Keep it under 80
  characters.") is real, correctly mapped, and reachable via direct API calls, but is
  structurally very hard to reach through the shipped UI itself, because `MessageInput.jsx`'s
  `<input maxLength={80}>` exactly mirrors the server's own 80-char cap (`moderation.json`
  `maxLength: 80`) — since `trimmedText.length` can never exceed the raw input's length, and the
  raw input can never exceed 80 via normal typing/paste, a real user can never trigger this
  specific server code through ordinary interaction (as opposed to `Create.jsx`'s title field,
  where the analogous "too long" case is folded into the *same* `invalid_title` code as "empty,"
  so no code there is orphaned). This is the same *class* of "client-side constraint quietly
  orphans a written, ratified error branch" bug that HM-5's own retry already caught once this
  task (the `invalid_message` disabled-button bug) — worth the orchestrator's attention as a
  product decision (loosen the input cap so a paste-over-limit can reach the server check and
  show the message, or accept it as API-only/defense-in-depth and leave as-is), not a code defect
  I'm fixing myself.
