## Findings reported to the orchestrator (not fixed directly — architectural/judgment calls)
1. **`backend/src/services/ogImage.js:383-391` (`getOgImage`) — cache-stampede on concurrent
   cache misses (verified empirically).** No in-flight de-duplication: on a cache miss, every
   concurrent request independently calls `renderOgImage` (CPU-bound Skia work). Verified with 15
   simultaneous requests to the same never-cached slug → 15 full renders, not 1. Failure scenario:
   a timer link goes viral and gets opened by many chat-app link-unfurlers at once right after the
   60s TTL expires (or on first-ever request) — server does N redundant renders instead of 1.
   Suggested fix: a "single-flight" in-flight-promise map keyed the same as the cache, so
   concurrent misses for the same key await one shared render.
2. **No automated test coverage for the OG-image feature.** `grep` across `backend/test/` finds
   zero references to `og-image`/`ogImage`/`ogRouter`; verification was curl/manual-only per the
   retro. Every other feature in this codebase (reactions, messages, moderation, rate-limit) has
   dozens of automated regression tests; this route has none. Failure scenario: a future refactor
   of `TimerService.getBySlug` or the cache logic silently breaks `/og-image.png` with zero signal
   from `npm test`. Suggested fix: assign automated tests (200 + PNG magic bytes for a valid slug,
   404 for unknown, ended-state variant, cache hit/miss via `clearOgImageCache()`) — likely folds
   into SC-6 (qa-engineer) or a follow-up for `share-card-developer`.

Both are reported as findings, not fixed directly, per Rule 2 (architectural caching-strategy and
test-ownership decisions belong to the orchestrator, not a same-pass fix).
