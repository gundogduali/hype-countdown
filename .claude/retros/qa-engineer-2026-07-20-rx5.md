## What the self-review found (and I fixed before finishing)
- My first ad-hoc Playwright pass reported false FAILs on the count display (`textContent`/`innerText` on the button showed only the emoji, not the number). Root cause: the count is rendered by `@number-flow/react`, a custom element with a **closed rendering pipeline via Shadow DOM** — plain DOM `textContent`/`innerText` never reflects shadow content, but the browser's accessibility tree does. Switched to `locator.ariaSnapshot()` to read the real rendered/announced value; re-ran and the feature was actually correct all along. This is the same *class* of mistake as my existing lesson (probing technique lying about a working feature) but a new root cause (shadow DOM, not timeout races) — documented as a fresh lesson below rather than assuming the existing one covered it.
- A second self-inflicted false signal: after killing what I thought was my backend and "restarting" it, a curl probe still returned `rate_limited` and my new-process log showed a DB error. Diagnosis: my `pkill -f "PORT=3097"` never matched anything, because an env-var prefix (`PORT=3097 nohup node ...`) is consumed by the shell and is not part of the exec'd process's argv — so it's invisible to `pkill -f`. The truly-old process kept running the whole time (with its rate limiter already exhausted from my own repeated test runs), while my "new" process crashed on a DB path I'd just deleted out from under a directory it needed. Fixed by killing the exact PID (from `ps`/`lsof`) instead of pattern-matching on env vars, and re-verified with a clean instance. No product bug — entirely my own tooling mistake — but recorded as a lesson since it wasted time and could otherwise produce a false PASS/FAIL under the wrong server state.

## Lessons learned (added to my agent file)
- `@number-flow/react` (and similar animated-number libs) render into a **Shadow DOM** — `textContent`/`innerText` on the host element will not see the number at all, causing a false FAIL that looks identical to "the count never updated." Use `locator.ariaSnapshot()` (or otherwise query the accessibility tree) to read the actual rendered/announced value for any component that might use a custom element with a shadow root.
- `pkill -f "SOME_ENV_VAR=value ..."` does **not** match processes started with a leading shell env-var assignment (`VAR=value cmd`), because the assignment is consumed by the shell and never appears in the exec'd process's argv/cmdline — it silently kills nothing. To kill a specific instance you started this way, find and kill the exact PID via `ps`/`lsof` instead of pattern-matching on the env var text.
